# docs-validation-hook 要件定義書

## 概要

Tsumiki（編集コマンド）実行時に、docs/rules配下のドキュメント読込み状況を監視し、必要な実装ルール文書を読まずに編集を行うことを防止するhookシステムの要件定義。

## ユーザストーリー

### ストーリー1: 実装ルール遵守の強制

- **である** Claude Code利用開発者 **として**
- **私は** Tsumiki編集コマンド実行前に必ず関連するdocs/rulesドキュメントを確認 **をしたい**
- **そうすることで** 実装ルール違反や品質問題を事前に防止できる

### ストーリー2: 効率的な開発フロー

- **である** 開発チーム **として**
- **私は** ルール確認を自動化してコード品質を維持 **をしたい**
- **そうすることで** レビュー工数削減と一貫性のある実装を実現できる

## 機能要件（EARS記法）

### 通常要件

- REQ-001: システムはTsumiki編集コマンド実行時にdocs/rules配下の読込み状況をチェックしなければならない
- REQ-002: システムは未読の必須ドキュメントが存在する場合、編集コマンドをブロックしなければならない
- REQ-003: システムは読込み必要なドキュメントのリストをユーザに表示しなければならない
- REQ-004: システムは読込み状況を一時的に記録・管理しなければならない

### 条件付き要件

- REQ-101: 任意のファイル編集の場合、システムは`docs/rules`配下の全てのドキュメント読込みを必須としなければならない
- REQ-102: 新しいドキュメントが`docs/rules`に追加された場合、システムは自動的に必須読込み対象に含めなければならない
- REQ-103: 全ドキュメント読込み済みの場合、システムは編集コマンドを許可しなければならない
- REQ-104: `docs/rules`配下のファイル構成が変更された場合、システムは読込み状況をリセットしなければならない

### 状態要件

- REQ-201: セッション開始時にある場合、システムは読込み状況をリセットしなければならない
- REQ-202: 複数のルールファイルが更新された場合にある場合、システムは再読込みを要求しなければならない

### オプション要件

- REQ-301: システムは読込み済みドキュメントのタイムスタンプ管理をしてもよい
- REQ-302: システムは特定のファイル種別に対するルール適用除外設定を提供してもよい

### 制約要件

- REQ-401: システムはhookの実行時間を3秒以内に制限しなければならない
- REQ-402: システムは読込み状況の記録をセッション内のみに限定しなければならない
- REQ-403: システムは`docs/rules`配下のファイル構成変更を動的に検知して状況をリセットしなければならない
- REQ-404: システムは`docs/rules`配下の全ての`.md`ファイルを動的に検索・対象化しなければならない

## 非機能要件

### パフォーマンス

- NFR-001: Hook実行時間は3秒以内であること
- NFR-002: ファイル読込み状況チェックは軽量であること（メモリ使用量最小化）

### セキュリティ

- NFR-101: 読込み状況の記録は一時的で、セッション終了時に削除されること
- NFR-102: ドキュメントの内容を外部に送信しないこと

### ユーザビリティ

- NFR-201: エラーメッセージは日本語で分かりやすく表示されること
- NFR-202: 必要なドキュメントのパスを明確に表示すること
- NFR-203: ドキュメント読込み後の再実行方法を案内すること

## Edgeケース

### エラー処理

- EDGE-001: docs/rulesディレクトリが存在しない場合→警告表示後、編集許可
- EDGE-002: 必須ドキュメントファイルが存在しない場合→エラー表示、編集拒否
- EDGE-003: ドキュメント読込み中にファイルが変更された場合→再読込み要求
- EDGE-004: hookスクリプト実行権限がない場合→権限エラー表示、編集拒否

### 境界値

- EDGE-101: 複数のドキュメントが同時に必要な場合の優先度処理
- EDGE-102: セッション境界での読込み状況の引き継ぎ
- EDGE-103: 大きなドキュメントファイルの処理時間制限

## 実装対象ファイルの動的検知

### 対象ファイル
- 全ての編集可能ファイル（プロジェクト配下の実装ファイル）
- 除外: 設定ファイル（`.claude/**`）、ドキュメント（`docs/**`）

### 必須ドキュメント
- `docs/rules`配下の全ての`.md`ファイルを動的に検索
- 現在存在するファイル: `frontend.md`, `styling-rule.md`, `msw-mock-system.md`, `rule-updating.md`
- 将来追加されるファイルも自動的に対象化

### 動的検知仕様
- hookスクリプト実行時に`find docs/rules -name "*.md"`でファイル一覧を取得
- ファイル構成の変更を検知した場合、読込み状況をリセット
- セッション中のファイル構成ハッシュと比較して変更を検知

## 受け入れ基準

### 機能テスト

- [ ] Tsumiki編集コマンド実行時にhookが正常に動作する
- [ ] 未読ドキュメント存在時に編集がブロックされる
- [ ] 全ドキュメント読込み後に編集が許可される
- [ ] 適切なエラーメッセージが表示される
- [ ] `docs/rules`配下の全ファイルが動的に検知される
- [ ] 新規ドキュメント追加時に自動的に必須対象となる
- [ ] ドキュメント削除時に読込み状況が適切に更新される

### 非機能テスト

- [ ] Hook実行時間が3秒以内である
- [ ] セッション終了時に読込み記録が削除される
- [ ] 複数回実行時のパフォーマンス劣化がない
- [ ] 日本語エラーメッセージが正しく表示される

### 統合テスト

- [ ] 既存のrules-validation.shとの共存が正常に動作する
- [ ] PreToolUseフックチェーンでの実行順序が適切である
- [ ] Claude Code設定ファイルでの有効化が正常に動作する

## 技術仕様

### 実装方式
- Bash scriptによるhook実装
- `.claude/settings.local.json`のPreToolUseフック設定
- 読込み状況の一時ファイル管理（`/tmp/.claude_docs_read_status`）

### 対象コマンド
- Edit, MultiEdit, Write等のファイル編集コマンド
- mcp__serena__replace_symbol_body等のserena編集コマンド

### 除外対象
- 読込み専用コマンド（Read, Grep, Find等）
- 設定ファイル（`.claude/**`）
- ドキュメントファイル（`docs/**`）
